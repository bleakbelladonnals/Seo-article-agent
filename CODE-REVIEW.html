项目概述

基于 CrewAI 的 B2B SEO 文章自动化流水线。输入关键词 + 产品品类 + 材质，4 个 AI Agent 协作完成：研究关键词 → 策划大纲 → 撰写文章（含 Schema/图片SEO/llms.txt）→ 7维评分 → 不通过自动修改（最多 5 轮）。

技术栈：DeepSeek V4 Pro（主模型）+ Qwen3-VL（图片分析）+ 本地 JSON 知识库（5 品类关键词 + 材质工艺数据）

显著亮点（8 项）

1. 自适应质量闭环

crew.py:274 的「写作 → 评审 → 定向修改 → 再次评审」循环是项目最大亮点。只重跑 Writer + Reviewer（而非 4 个 Agent），提取未通过维度的修改建议精准注入，全自动无需人工干预。

2. 关注点分离清晰

config.py（集中参数）、agents.py（角色定义）、tasks.py（Prompt 指令 + 依赖链）、crew.py（编排逻辑）、main.py（入口）。每个文件职责单一，均不超过 400 行。

3. AI SEO 覆盖完整

Schema（Article / FAQPage / ImageObject 三种 JSON-LD）、llms.txt 生成、alt 文本 + webp + 文件名规范、作者署名（EEAT 信号）、来源引用的统计数据、AI 可见性检查清单——全部由 Writer 一次性生成，比大多数生产环境的 SEO 流水线更全面。

4. 知识库优雅降级链

图片分析 → 品类提取 → 知识库匹配 → LLM 经验估算。流水线在 crew.py:188-234 逐级退化：视觉模型不可用时用关键词猜测品类，知识库无匹配时 Researcher 自行估算，没有对任何单一组件的硬依赖。

5. H2 内容禁令 + 改写范例

tasks.py:79-89 明确禁止 MOQ、交期、付款、认证、价格、物流等商业术语出现在 H2/H3/FAQ 标题中，并给出具体改写示例（"BAD → GOOD"）。有效阻止 AI 文章退化为商业垃圾。

6. E-E-A-T 一票否决

Reviewer 检测到编造数据直接给 EEAT=0 分（tasks.py:193），此为硬门槛：即使其他维度全部通过，EEAT=0 整篇文章也不通过。配合 Writer 的 "According to..." 来源引用要求，形成了真正的执行约束。

7. 知识库含真实工程数据

keywords.json 包含 ASTM B86 密度、ISO 6506 硬度、电镀厚度公差、珐琅固化温度等工业参数——不是空洞的关键词堆砌。Writer 可以基于这些真实数据写出 "which means..." 利益桥接。

8. 无平台锁定

纯 Python，不依赖 Dify/Coze/LangChain（仅 CrewAI）。换模型只需改 .env 一行。OpenAI 兼容客户端模式（crew.py:66-67）意味着任何厂商（OpenAI / DeepSeek / Qwen / GLM）都能即插即用。

高危问题（2 项）

高危
逻辑缺陷 修订循环中 Writer 丢失 Research/Planning 上下文
writing_task 声明了 context=[research_task, planning_task]（tasks.py:179），即 Writer 依赖研究员和策划的输出。但在修订轮中，Crew 只包含 [writer, reviewer]——Research 和 Planning 任务不在 Crew 中，CrewAI 的 context= 无法解析这些输出。修订模式下的 Writer 仅通过 inputs= 收到 {fix_suggestions}，拿不到原始研究数据。如果模型没有忠实记住原文上下文，改写可能跑偏。修复方式：在修订轮中将首轮的研究和策划输出显式注入 Writer 的输入变量。
位置：crew.py:286-291 · tasks.py:179
高危
安全/稳定性 API 密钥明文存储 + 客户端无超时/重试
.env 中 API 密钥以明文存储。config.py 加载时不做校验——缺失的 Key 静默变为 None。crew.py:67 初始化的 OpenAI 客户端没有设置 timeout、max_retries、也没有错误分类处理。网络抖动或 API 限流（429）会直接导致流水线崩溃。DeepSeek 和 Qwen 的 API 在连续调用 4 个 Agent 时很容易触发频率限制。修复方式：给 OpenAI 客户端加 timeout=60, max_retries=3；config.py 启动时校验必填 Key，缺失即抛明确错误。
位置：config.py:12-18 · crew.py:67-68 · .env

中危问题（6 项）

中危
健壮性 _extract_scorecard() 的 JSON 提取逻辑脆弱
用 text.rfind("}") 找最后一个右花括号（crew.py:353）。如果模型在 JSON 后面附加了文字说明（比如 "这是我的评审结果：{...} 希望对你有帮助！"），rfind 会定位到错误的括号。如果 JSON 包含嵌套对象（dimensions 数组内的对象），首尾花括号切片可能包含多余内容或截断。改用括号深度计数器会更稳健，或者用更精确的正则锚定。
位置：crew.py:342-361
中危
健壮性 _extract_image_seo() 段落解析不可靠
通过全小写子串匹配 "image seo" 或 "alt text" 来检测段落起点（crew.py:383），容易被误触发（文章正文提到 "image SEO best practices"）。段落边界用空行判定（crew.py:396），一旦 LLM 输出的 markdown 表格中间有换行就会提前终止。更稳健的做法：精确匹配 "## Image SEO Data" 标题，解析到下一个 "## " 标题为止。
位置：crew.py:376-397
中危
设计 LLM 直出 Schema 存在天然风险
Writer 以 Markdown 代码块形式直接输出 JSON-LD。如果 LLM 幻觉出错误属性名或产生非法 JSON，搜索引擎的结构化数据解析会完全失败。生产环境应该用 Python 函数构造 Schema（如 build_article_schema()），传入已验证的字段值，确保输出的 JSON-LD 语法合法且属性值正确。HANDOFF.md 已将此列为已知缺口，但值得再次强调：一条非法 Schema 会导致页面全部结构化数据失效。
位置：tasks.py:150-154 · HANDOFF.md:169
中危
工程 全项目零测试
没有 tests/ 目录，没有 pytest 文件，没有 doctest。crew.py 中每一个工具函数（_extract_scorecard、_build_fix_text、_retrieve_keywords、_extract_image_seo）都可以通过 mock API 调用和 fixture 数据做隔离测试，但目前一个都没有。考虑到这些函数处理的是 LLM 输出的非结构化文本（天然不稳定），缺乏测试意味着每次改代码都在冒险。
位置：crew.py（全部 _ 函数）· 无 tests/ 目录
中危
设计 _build_fix_text() 只收集 score < 6 的维度
Reviewer 对 7 个维度打分（1-10），理论上任何未满分维度都可能附带修改建议。但 _build_fix_text()（crew.py:369）只收集 score < 6 的建议。如果一个维度得 6 分但 Reviewer 给了有用的修改建议，这些建议会被静默丢弃。而通过线是 48/70（平均约 6.86），得 6 分的维度实际低于通过所需的平均水平。修改建议的收集阈值应该与通过阈值对齐。
位置：crew.py:364-373 · config.py:30
中危
工程 全项目用 print() 代替日志系统
流水线所有状态输出都靠 print()。没有日志级别（debug/info/warning/error）、没有时间戳、没有文件输出、没有 Token 用量统计。出问题时无法按级别检索日志、无法追踪单次运行的资源消耗。最简修复：引入 Python logging 模块 + 轮转文件 Handler。另外应当加入 Token 用量统计——对一个最多跑 5 轮修改的系统，成本追踪是刚需。
位置：crew.py（全部 print() 调用）· main.py:49

低危问题 / 改进建议（4 项）

低危
getattr(args, "type", "") 规避 Python 内置名冲突
main.py:80 用 getattr(args, "type", "") 是因为 args.type 与 Python 内置 type 冲突。更干净的修复：把参数名改为 --customer-type，用 args.customer_type 取值，无需 getattr 迂回。
位置：main.py:72, 80
低危
CLI 模式缺少 --images 参数
交互模式支持上传图片，但 CLI 模式没有 --images 参数，导致自动化/脚本化运行时无法使用图片分析功能。修复很简单：加一行 parser.add_argument("--images", nargs="*", default=[])。
位置：main.py:70-73 · HANDOFF.md:170
低危
.format() 与 CrewAI 模板变量共存，容易踩坑
review_task.description 在 tasks.py:211 用 .format(threshold=QUALITY_THRESHOLD) 注入通过分数线，而 writing_task.description 中的 {fix_suggestions} 是 CrewAI 模板变量。目前能正常工作只是因为 .format() 会忽略不认识的占位符……但前提是 CrewAI 恰好用 {var} 语法。如果后续维护者新增的 .format() 键名恰好和 CrewAI 模板变量重名，会无声崩溃。至少应加一行注释警告。
位置：tasks.py:105, 211
低危
知识库 Volume/KD 数据无过期机制
keywords.json 硬编码了搜索量和难度值。这些数据会衰减：Google 每月更新关键词数据，竞争格局也在变化。目前没有任何刷新策略（Ahrefs/Semrush API 拉取、定期人工复核、或置信度衰减）。对生产环境而言，3-6 个月后这些数字就会产生误导。
位置：knowledge/keywords.json

改进建议（按优先级排序）

#改进项工作量影响

面试准备提示

面试官大概率会追问的点（以及如何回应）：

  1. "Schema 为什么是 LLM 生成的而不是代码拼接？" → 承认这是当前版本的权衡（快速验证可行性），但已规划了 Python 构造函数方案（改进项 #8），且目前输出质量通过 Reviewer 的 JSON 校验做了基本把关。
  2. "知识库只有 5 个品类，怎么扩展？" → 当前是 MVP 验证阶段。扩展路径：接入 Ahrefs/Semrush API 自动刷新 Volume/KD 数据，或用 Dify 向量检索（原版能力）作为升级方案。
  3. "修改循环最多 5 轮，如果一直不过怎么办？" → 5 轮后仍输出最终版本并附评分记录（crew.py:313-325 保存 scorecard），不会无限循环。实际运行中大部分文章 1-2 轮通过。
  4. "为什么不用 Dify 的向量检索？" → 刻意去掉平台绑定。本地 JSON 的缺点是检索精度下降，但优点是零依赖、可离线运行、数据完全受控。面试时可以主动反问对方对平台绑定 vs 自研的看法。
